## ⚠️ Major spec finding: Copper Golem already exists in vanilla 26.1.2

`CarvedPumpkinBlock.trySpawnGolem` checks **three** vanilla patterns: snow → iron → **copper**. A vanilla `CopperGolem` class lives at `net.minecraft.world.entity.animal.golem.CopperGolem` and is created by **single copper block + carved pumpkin** (1-tall pillar, accepting any block tagged `#minecraft:copper`). On spawn it replaces the copper block underneath with a copper chest and has weathering states.

**Decision (per user, see brainstorming transcript 2026-05-15):** Both copper paths coexist.
- Single copper block + pumpkin → vanilla copper golem (chest-bearer)
- T-shape of copper blocks + pumpkin → MultiGolem's copper variant (attachment on `IronGolem`)

Our mixin into `CarvedPumpkinBlock.trySpawnGolem` injects at HEAD and checks for a T-shape match in our supported body materials (copper/gold/emerald/diamond/netherite) **before** vanilla runs. If we match a T-shape, we spawn the variant golem and cancel. If not, vanilla logic runs and may match one of its three patterns (including single-block copper).

## LivingEntity

- **FQN:** `net.minecraft.world.entity.LivingEntity`
- **`getMaxHealth()`** at line 1960 — `public **final** float` — calls `getAttributeValue(Attributes.MAX_HEALTH)`. **Final → cannot mixin directly.**
- **`getAttributeValue(Holder<Attribute>)`** at line 2169 — `public double` (NOT final). Body is `this.getAttributes().getValue(attribute)`. Can be mixed in.
- **`setLastHurtByMob(LivingEntity)`** at line 638 — `public void`. Anger-cancellation target. Mix into `LivingEntity.class` and filter to `IronGolem` instances at runtime (see "Inherited methods" note below).
- **Hot-path check:** `getAttributeValue` is called from `getMaxHealth` (used by rendering, `getCrackiness`, healing cap logic), `getMovementSpeed` (AI/pathfinding every tick), `getKnockbackResistance` (combat), and more. Many calls per entity per tick.

## Decisions

### 1. Stats strategy → **Plan B: deterministic transient modifiers**

**Rationale:** `getAttributeValue` is called many times per entity per tick from hot paths (rendering's hp-bar lookup, AI's per-tick movement speed read, `getCrackiness` reading max health for every iron golem each tick). Mixing in there adds an attachment lookup to every call. Transient attribute modifiers use vanilla's native attribute machinery and have zero per-read cost — the modifier is added once when the variant attaches, and `getAttributes().getValue(attr)` includes it for free.

**Implementation sketch** (for Task 6):
- When variant is set or loaded on an `IronGolem`, apply two transient modifiers: HEALTH delta (= variant.maxHealth − 100), DAMAGE delta (= variant.attackDamage − 15).
- Use stable `ResourceLocation` IDs: `multigolem:variant_health`, `multigolem:variant_attack_damage`.
- Idempotent re-apply on tick (cheap O(1) check via `instance.getModifier(id)`).
- On entity load, also clamp `setHealth(min(currentHealth, newMaxHealth))` — handles the "config max_health was lowered" case.

### 2. Drops strategy → **Plan A: loot-table override with custom condition**

**Rationale:** Vanilla iron golem death loot comes from `data/minecraft/loot_tables/entities/iron_golem.json` (via `EntityType.IRON_GOLEM`'s default loot table). `LootContextParams.THIS_ENTITY` is stable vanilla API and exposes the killed entity to our custom `LootItemCondition`. Mod compatibility is friendly — other mods using Fabric's `LootTableEvents.MODIFY` will coexist (or we can use `LootTableEvents.MODIFY` ourselves rather than replacing the file outright).

**Implementation sketch** (for Task 10):
- Register `LootItemConditionType` with id `multigolem:has_golem_variant`, holding a `GolemVariant` field.
- The condition reads `ctx.getParamOrNull(LootContextParams.THIS_ENTITY)`, calls `GolemVariantAttachment.get(entity)`, compares to its variant field.
- Prefer Fabric's `LootTableEvents.MODIFY` over file replacement for `minecraft:entities/iron_golem` — appends our 5 conditional pools (one per non-iron variant) without disturbing vanilla's iron drops or any other mod's modifications.

## Spec corrections required

These were uncovered during the spike and (in some cases) during Task 3 build verification:

0. **`ResourceLocation` was renamed to `Identifier` in MC 26.1.2.** Class is now `net.minecraft.resources.Identifier` with static factory `Identifier.fromNamespaceAndPath(namespace, path)`. The plan and the spike's first draft both used `ResourceLocation` — every mixin and registration file needs `Identifier` instead. (Yes, this is the Yarn-era name — Mojang adopted it as the official name in 26.1.)
1. **Iron golem vanilla ATTACK_DAMAGE is 15.0, not 14.0.** Update `MultiGolemConfig.defaults()` and the spec §6.1 table — Iron row should read `attack_damage = 15.0`, average swing 7.5–22.5 (matches what the plan's narrative already says).
2. **IronGolem package is `.animal.golem`, not `.animal`** — every mixin and import in subsequent tasks needs `net.minecraft.world.entity.animal.golem.IronGolem`.
3. **Spawn reason is `EntitySpawnReason.TRIGGERED`**, not `MOB_SUMMONED`.
4. **`trySpawnGolem` is private** — mixin targets a private method (mixin handles this fine, but worth noting we can't call it from outside).
5. **Item consumption** in mobInteract uses `ItemStack.consume(1, player)`, not `shrink(1)`. The `consume` method handles creative-mode logic internally — don't manually check `getAbilities().instabuild`.
6. **`hurtServer` replaces `hurt`/`actuallyHurt`** in 26.1.2 for server-side damage handling.
7. **Copper golem is vanilla** — see top section. Spec needs an addendum noting this coexistence design.
8. **Reuse `CarvedPumpkinBlock.clearPatternBlocks`** in our creation handler instead of rolling our own ham-fisted block clearer (the plan's draft code clears blocks manually with broad neighbor coverage).
9. **Mixin `@Inject` does NOT walk the class hierarchy** — see the "Inherited methods" section below. Earlier drafts of this doc said `setLastHurtByMob` and `tick` could be mixed into `IronGolem.class`; that's wrong and crashes server startup with `Critical injection failure: ... could not find any targets matching '<method>' in net/minecraft/world/entity/animal/golem/IronGolem`.

## Inherited methods — mixin gotcha

**Rule:** `@Inject` only validates against methods **declared on the target class**. It does NOT walk the superclass chain. If the method isn't in IronGolem's own source, mixing into `IronGolem.class` will fail at apply-time with `Critical injection failure: ... could not find any targets matching '<method>' in net/minecraft/world/entity/animal/golem/IronGolem`.

**How to handle each case:**

- **Target is declared on `IronGolem`** (e.g. `mobInteract`, `die`): mix into `IronGolem.class` directly. ✓
- **Target is inherited from `LivingEntity` / `Mob` / `Entity`** (e.g. `tick`, `setLastHurtByMob`, `aiStep`): mix into the **declaring class** and filter to `IronGolem` instances at runtime: `if (!(((Object) this) instanceof IronGolem self)) return;`. Be aware this fires for *every* `LivingEntity`, so keep the filter at the very top and the body cheap.
- **Logic only needs to run on attachment change / entity load** (e.g. applying variant attribute modifiers): skip the mixin entirely. Use `ServerEntityEvents.ENTITY_LOAD` plus a call site in `GolemVariantAttachment.set()`. See `attribute/VariantAttributes.java`.

Don't trust the "No refMap loaded" line in the error log — in this project the production runtime is Mojang-named throughout, so no refmap is needed. The real cause when you see that error is almost always the inherited-method issue above.

- **`getTextureLocation` signature:** `public Identifier getTextureLocation(IronGolemRenderState state)` — **takes state, not entity**, exactly as the V2 plan assumes. Inject at HEAD with `cancellable = true` and `setReturnValue` to override.

### Spike 2 — `ServerLivingEntityEvents.ALLOW_DAMAGE` semantics

Verified by reading `net/fabricmc/fabric/api/entity/event/v1/ServerLivingEntityEvents.java` in `fabric-entity-events-v1-2.1.12+14b92d8987-sources.jar`:

- **Listener interface signature:** `boolean allowDamage(LivingEntity entity, DamageSource source, float amount)`
- **Event factory:** `EventFactory.createArrayBacked(AllowDamage.class, ...)` — listeners fire in registration order.
- **Short-circuit on false: YES.** Factory loops listeners; first `false` returns immediately and skips later listeners.
- **Fired BEFORE armor / mitigation:** javadoc says "fired from `LivingEntity.hurtServer`, before armor or any other mitigation are applied." Returning `false` cancels the entire damage application path, so no `lastHurtByMob`, no aggro, no death trigger, no advancement.
- **Per-strike for `LIGHTNING_BOLT`:** fires once per `hurtServer` call; `LightningBolt` calls `hurt` once per struck entity per strike. No multi-fire concern.
- **Implication for §5.1 copper atomic cancel+heal:** safe. Single listener returning false guarantees no side effects; heal + particles happen in the same listener body before the return.

### Spike 6 — Mob targeting hook (CHOSEN)

**Critical context:** Vanilla iron golem at `IronGolem.registerGoals()` line 76 uses:

```java
this.targetSelector.addGoal(3, new NearestAttackableTargetGoal(this, Mob.class, 5, false, false,
    (target, var1) -> target instanceof Enemy && !(target instanceof Creeper)));
```

So vanilla iron golems already exclude creepers from PROACTIVE targeting. But `HurtByTargetGoal` at priority 2 makes the golem retaliate against ANYTHING that hit it — including creepers. To enforce `ignored_target_types` on both paths (proactive + reactive), we need a unified veto.

**Chosen hook: option (b) — `Mob#setTarget(LivingEntity)`.**

- **FQN:** `net.minecraft.world.entity.Mob`
- **Method:** `public void setTarget(@Nullable LivingEntity target)` at line 251
- **Descriptor:** `(Lnet/minecraft/world/entity/LivingEntity;)V`
- **Mixin pattern:** `@Mixin(Mob.class) @Inject(method = "setTarget", at = @At("HEAD"), cancellable = true)` — filter to `IronGolem` at runtime (per V1 inherited-method pattern). Cancel if target is excluded by the golem's tier's `ignored_target_types`.

**Rejected alternatives:**

- **(a) `NearestAttackableTargetGoal.canUse()`:** would only catch proactive targeting, missing `HurtByTargetGoal` retaliation. Two mixins required to cover both — more invasive than (b).
- **(a-alt) `TargetingConditions.test(...)`:** shared across many goals/mobs, would fire for every entity targeting decision in the game and require runtime filtering anyway — no cleaner than (b).
- **(c) per-tick `setTarget(null)`:** allows one-tick flicker; rejected per Codex round-3 guidance.

## V3 Spike Findings (2026-05-17)

Output of V3 plan Task 1. Used by V3 village natural-spawn weighting.

### Villager help-call spawn target

- **Chosen target:** `net.minecraft.world.entity.npc.villager.Villager`.
- **Named on `Villager.class`: yes.** The villager golem help-call logic is still a named method on `Villager` itself, not moved into a behavior/task helper.
- **Method signature:** `public void spawnGolemIfNeeded(final ServerLevel level, final long timestamp, final int villagersNeededToAgree)`.
- **Mixin descriptor:** `(Lnet/minecraft/server/level/ServerLevel;JI)V`.
- **Source evidence:** `Villager.java` line 835 declares `spawnGolemIfNeeded`; line 841 calls `SpawnUtil.trySpawnMob`; line 845 notifies nearby villagers with `GolemSensor::golemDetected` only when the returned `Optional` is non-empty.
- **Spawn reason:** `EntitySpawnReason.MOB_SUMMONED` is the actual villager help-call spawn reason in 26.1.2. The V3 spec's pre-spike assumption is confirmed for this path.
- **Call-site arguments:** `SpawnUtil.trySpawnMob(EntityType.IRON_GOLEM, EntitySpawnReason.MOB_SUMMONED, level, this.blockPosition(), 10, 8, 6, SpawnUtil.Strategy.LEGACY_IRON_GOLEM, false)`.

### `SpawnUtil.trySpawnMob`

- **FQN:** `net.minecraft.util.SpawnUtil`.
- **Method signature:**
  ```java
  public static <T extends Mob> Optional<T> trySpawnMob(
      EntityType<T> entityType,
      EntitySpawnReason spawnReason,
      ServerLevel level,
      BlockPos start,
      int spawnAttempts,
      int spawnRangeXZ,
      int spawnRangeY,
      SpawnUtil.Strategy strategy,
      boolean checkCollisions
  )
  ```
- **Mixin descriptor:** `(Lnet/minecraft/world/entity/EntityType;Lnet/minecraft/world/entity/EntitySpawnReason;Lnet/minecraft/server/level/ServerLevel;Lnet/minecraft/core/BlockPos;IIILnet/minecraft/util/SpawnUtil$Strategy;Z)Ljava/util/Optional;`.
- **Return type:** `java.util.Optional<T>`. `SpawnUtil.java` line 45 returns `Optional.of(mob)` after `level.addFreshEntityWithPassengers(mob)` and `mob.playAmbientSound()`. Line 53 returns `Optional.empty()` after all attempts fail.

### Spawned-golem capture strategy

- **Chosen strategy: Candidate B, plain Mixin `@Redirect` on the `SpawnUtil.trySpawnMob` call inside `Villager#spawnGolemIfNeeded`.**
- **Why chosen:** the call returns `Optional<IronGolem>` directly at the narrow villager help-call site. A redirect can call `SpawnUtil.trySpawnMob` with the original arguments, inspect the returned `Optional`, apply V3 only when a golem was actually spawned, and return the same `Optional` so vanilla's `GolemSensor::golemDetected` behavior is preserved.
- **Mixin Extras required:** no. Use core Mixin `@Redirect`; do not add Mixin Extras for V3 unless a later source change invalidates this redirect.
- **Candidate A rejected:** the `Optional` return is used inline in `if (!SpawnUtil.trySpawnMob(...).isEmpty())` rather than stored in a visible local variable, so a TAIL local capture is more fragile and likely unavailable.
- **Candidate C rejected:** a thread-local plus global entity-load/event fallback is broader than needed, relies on ordering/global state, and could accidentally catch unrelated golems if the flag leaked.

### Scope safety

- The hook remains inside `Villager#spawnGolemIfNeeded`, so it is limited to villagers that have already met vanilla's golem-help conditions.
- Mob spawners are excluded because they do not execute this villager method.
- Spawn eggs are excluded because they do not execute this villager method.
- Commands are excluded because `/summon minecraft:iron_golem` does not execute this villager method.
- Existing golems are excluded because the hook only runs at the successful villager spawn call site.
- The villager path does not call `IronGolem#setPlayerCreated(true)`. V3 must not add that call; village variants remain natural defenders with `playerCreated=false`.

### V2 loot composition

- `HasGolemVariantLootCondition` reads the killed entity at death time via `LootContextParams.THIS_ENTITY`.
- The condition then calls `GolemVariantAttachment.get(e)` and compares it to the configured variant.
- `MultiGolem.registerVariantLoot()` appends conditional pools to `EntityType.IRON_GOLEM`'s default loot table for Copper, Gold, Emerald, Diamond, and Netherite.
- Therefore a V3 spawn-time attachment on the villager-spawned `IronGolem` composes with V2 loot: by the time the entity dies, the loot condition reads the same attachment and drops the variant-specific items.

### Player-built T-pattern creation attribution

- **Current pumpkin hook:** `CarvedPumpkinBlock.trySpawnGolem(Level, BlockPos)` has no player parameter.
- **Chosen strategy:** add a narrow `BlockItem.place(BlockPlaceContext)` redirect around `placeBlock(updatedPlaceContext, placementState)` for carved pumpkin and jack-o-lantern placements. Store the current `ServerPlayer`, `ServerLevel`, and exact pumpkin position in a thread-local only while the vanilla placement call is executing.
- **Why chosen:** placement and `CarvedPumpkinBlock.onPlace(...)` are synchronous, so the existing `CarvedPumpkinBlockMixin` can read the responsible player for the exact top position without a broad global fallback.
- **Rejected strategy:** do not scan nearby players or use a broad entity/event fallback; that would misattribute non-player or unrelated golem creation.

### Denial messages and healing denial

- **Message API:** use `Player.sendOverlayMessage(Component.literal(message))` for short denial messages. `sendOverlayMessage` is declared on `net.minecraft.world.entity.player.Player`, so `sendHealDenied(Player, ...)` is valid.
- **Healing denial result:** return `InteractionResult.FAIL` before healing, sound, or item consumption.

## V4 Spike Findings (2026-05-17)

Output of V4 spawn egg research. Used by marked iron-golem spawn egg design.

### `SpawnEggItem` internals

- **FQN:** `net.minecraft.world.item.SpawnEggItem`.
- **Superclass:** `SpawnEggItem` extends `Item`, but the relevant right-click entry points are declared directly on `SpawnEggItem`.
- **Block right-click method:** `useOn(UseOnContext)`.
- **Air/liquid right-click method:** `use(Level, Player, InteractionHand)`.
- **Descriptors:**
  - `useOn`: `(Lnet/minecraft/world/item/context/UseOnContext;)Lnet/minecraft/world/InteractionResult;`.
  - `use`: `(Lnet/minecraft/world/level/Level;Lnet/minecraft/world/entity/player/Player;Lnet/minecraft/world/InteractionHand;)Lnet/minecraft/world/InteractionResult;`.
  - `getType`: `(Lnet/minecraft/world/item/ItemStack;)Lnet/minecraft/world/entity/EntityType;`.
- **Shared spawn path:** both normal block placement and source-liquid placement call the private static helper `spawnMob(@Nullable LivingEntity user, ItemStack itemStack, Level level, BlockPos spawnPos, boolean tryMoveDown, boolean movedUp)`.
- **Entity creation call:** `spawnMob` reads `EntityType<?> type = getType(itemStack)` and then calls `type.spawn((ServerLevel) level, itemStack, user, spawnPos, EntitySpawnReason.SPAWN_ITEM_USE, tryMoveDown, movedUp)`.
- **Spawn reason:** normal spawn egg placement uses `EntitySpawnReason.SPAWN_ITEM_USE`.
- **Redirect target descriptor:** `Lnet/minecraft/world/entity/EntityType;spawn(Lnet/minecraft/server/level/ServerLevel;Lnet/minecraft/world/item/ItemStack;Lnet/minecraft/world/entity/LivingEntity;Lnet/minecraft/core/BlockPos;Lnet/minecraft/world/entity/EntitySpawnReason;ZZ)Lnet/minecraft/world/entity/Entity;`.
- **Entity type storage:** `SpawnEggItem` does not hold its own entity type reference. It reads `DataComponents.ENTITY_DATA` from the `ItemStack`; the vanilla `minecraft:iron_golem_spawn_egg` stack points at `EntityType.IRON_GOLEM`.
- **Stack components:** `EntityType.spawn(ServerLevel, ItemStack, LivingEntity, BlockPos, EntitySpawnReason, boolean, boolean)` applies item stack components through `applyComponentsFromItemStack(itemStack)` before the spawned entity is added to the level.
- **Spawner branch:** `useOn` has a separate spawner path that calls `BaseSpawner.setEntityId(type, ...)` directly and does not call `spawnMob`. A normal `spawnMob` redirect will not make a marked egg configure a spawner with a MultiGolem variant.

### Data component registry sync safety

- **Confirmed risk:** Fabric Registry Sync marks `BuiltInRegistries.DATA_COMPONENT_TYPE` with `RegistryAttribute.SYNCED`.
- **Implication:** registering a custom `DataComponentType` can create vanilla-client compatibility risk during registry sync. V4 should not add `multigolem:*` component types.
- **Safe path:** the decided `minecraft:custom_data` marker uses the existing vanilla `DataComponents.CUSTOM_DATA` type, so it avoids the custom registry-entry risk.
- **Design impact:** this contradicts the pre-spike expectation that custom data component registration is unsynced, but it reinforces the V4 constraint to use vanilla `minecraft:custom_data`.

### Spawn egg color and texture rendering

- **No `SpawnEggItemRenderer` source found:** 26.1.2 does not appear to use a dedicated `net.minecraft.client.renderer.item.SpawnEggItemRenderer`.
- **No runtime tint colors found on `SpawnEggItem`:** spawn eggs are no longer rendered from two color values read by `SpawnEggItem`.
- **Vanilla model path:** the item definition for `minecraft:iron_golem_spawn_egg` points at model `minecraft:item/iron_golem_spawn_egg`, and that model uses `parent: minecraft:item/generated` with `layer0: minecraft:item/iron_golem_spawn_egg`.
- **Hook shape:** per-variant spawn egg visuals need a client item-model or item-property hook, not the existing `IronGolemRenderer#getTextureLocation` mixin pattern. Entity renderer texture hooks do not affect inventory item icons.
- **Likely data-driven option:** a vanilla-safe `minecraft:custom_model_data` component can be added to the marked stack and used by a client-side item model `select` property to choose Copper, Gold, Emerald, Diamond, and Netherite generated models/textures.
- **Alternate option:** 26.1.2 also has a `minecraft:component` select item-model property that can inspect existing components such as `minecraft:custom_data`, but exact `CompoundTag` case matching may be more brittle than a dedicated custom-model-data string.
- **Implementation spike still needed:** before writing code, verify the exact 26.1.2 item-model JSON shape for overriding a vanilla item definition with a `select` property in Fabric resources.

### Recommended V4 source targets

- **Server spawn hook:** use a narrow redirect on the `EntityType.spawn(...)` invocation inside `SpawnEggItem.spawnMob`.
- **Thread-local:** not needed for normal spawn egg use because the redirect receives `ItemStack`, `LivingEntity user`, spawn position, spawn reason, and the returned entity from the exact spawn call.
- **First action:** the redirect should inspect the `minecraft:custom_data` `multigolem.variant` marker before any MultiGolem-specific permission, mutation, or messaging. If no marker is present, call vanilla `EntityType.spawn(...)` unchanged.
- **Permission timing:** the redirect has `user` and `itemStack` before spawning, so it can check the existing `multigolem.create.<variant>` node before calling vanilla spawn. A denied use can return `null`; vanilla `spawnMob` only consumes the egg when the returned entity is non-null.
- **Attachment timing:** after calling vanilla `EntityType.spawn(...)`, inspect the returned entity and attach/apply the selected `GolemVariant` only when it is an `IronGolem` created from a marked iron-golem egg.
- **Spawner behavior decision:** if V4 needs marked eggs to configure spawners as variants, that is a separate hook from normal egg use. Otherwise document that spawner configuration remains vanilla iron-golem behavior.
